## Methodology

- **Task:** Build a minimal Express API with JWT auth middleware and Redis rate limiter.
- **Agent A:** No Oversight. Standard coding tools only (write_file, read_file, run_terminal_cmd, list_dir).
- **Agent B:** Oversight enabled. Must call `oversight_get_by_path(path)` before editing auth.ts, auth-middleware.ts, rate-limiter.ts, or rateLimit.ts. Receives pre-seeded constraints (8 BASE_RECORDS) from the Oversight DB.
- **Evaluation:** Violation checker applies 13 constraints (auth + rate) from swe-bench evaluator. Each violation maps to a documented real-world incident cost (from swe-bench run-eval).

## Violation Breakdown

| ID | Title | Incident cost | Agent A | Agent B |
|----|-------|---------------|---------|---------|
| auth-004 | Rejects tokens with algorithm=none (alg:none attack) | $150k | 2/2 runs | 0/2 runs |
| rate-004 | Rate limit key includes route path (per-endpoint isolation) | $20k | 0/2 runs | 1/2 runs |

**auth-004:** jwt.verify() called without `{ algorithms: ['HS256'] }`; unsigned tokens with alg:none bypass signature verification.  
**rate-004:** Rate limit key uses IP only; all endpoints share one counter (cross-endpoint bleed).

## Interpretation

1. **Oversight reduces critical auth violations.** Agent A violated auth-004 in both runs; Agent B avoided it in both (auth constraints were queried before editing).
2. **Some variance remains.** Agent B introduced rate-004 in one run, showing oversight is not perfect but improves outcomes.
3. **Cost framing is shareable.** "$150k at risk vs $10k" communicates impact better than "1 violation vs 0.5."
4. **Query-by-default works.** Agent B made 2 oversight calls per run, retrieving constraints before editing relevant files.
